# Vector Databases for RAG

## What are Vector Databases?

Vector databases are specialized databases designed to store, index, and query high-dimensional vectors (embeddings). In the context of RAG (Retrieval-Augmented Generation), these databases store text embeddings - numerical representations of text that capture semantic meaning. They enable efficient similarity search, which is crucial for finding relevant documents to provide as context to language models.

Embedding vector databases are fundamentally different from traditional databases:
- **Traditional databases**: Store structured data (rows/columns), use exact match or SQL queries
- **Vector databases**: Store high-dimensional vectors, use similarity search (cosine, dot product)
- **Use case**: Find semantically similar content, not exact matches

## Key Concepts

### Embeddings
Embeddings are numerical representations of text, images, or other data in a high-dimensional vector space. In RAG systems, text embeddings are generated by models like nomic-embed-text-v1.5, Sentence-BERT, or OpenAI's text-embedding models.

Key properties:
- **Dimensionality**: Typically 768-1536 dimensions for text models
- **Semantic similarity**: Similar meanings have similar vectors
- **Distance preservation**: Semantic distance ≈ vector distance

### Vector Similarity
Common distance metrics for text embeddings:
- **Cosine Similarity**: Measures angle between vectors (0-1), most common for text
- **Euclidean Distance**: Straight-line distance between vectors
- **Dot Product**: Similarity measure often used in neural networks

Cosine similarity is preferred for text because it focuses on direction (semantic meaning) rather than magnitude.

### Indexing
Vector databases use specialized indexes to speed up similarity search. Without indexing, you'd need to calculate similarity with every vector (brute force), which is O(n) and slow for large datasets.

Common index types:
- **HNSW (Hierarchical Navigable Small World)**: Graph-based index, fast and accurate. Most popular choice.
- **IVF (Inverted File Index)**: Clustering-based index, good balance of speed/memory
- **PQ (Product Quantization)**: Compression technique, reduces memory usage

**Why indexing matters**: For 10,000 vectors with 768 dimensions, brute force similarity calculation takes ~10ms per query. With HNSW index, it takes ~1ms.

## Popular Vector Databases for RAG

### Qdrant
- Open-source, written in Rust
- High performance with HNSW indexing
- Easy to deploy with Docker
- Good for production use

### Pinecone
- Managed service (no self-hosting)
- Excellent performance
- Easy to get started
- Pricing based on usage

### Weaviate
- Open-source with GraphQL API
- Built-in vectorization
- Multi-modal support

### Milvus
- Open-source, scalable
- Supports multiple index types
- Good for large-scale deployments

### Chroma
- Simple and lightweight
- Easy integration with LangChain
- Good for prototyping

## Integration with RAG

### Typical Workflow
1. Chunk documents into smaller pieces
2. Generate embeddings for each chunk
3. Insert vectors into vector database
4. Query with user question embedding
5. Retrieve top-k most similar chunks
6. Use retrieved chunks as context for LLM

### Hybrid Search
Combine vector search with keyword search (BM25) for better results:
- **Vector search**: Captures semantic similarity (e.g., "car" ≈ "automobile")
- **Keyword search**: Captures exact matches (e.g., "car" matches "car")
- **RRF (Reciprocal Rank Fusion)**: Combines scores from both methods

## Best Practices

### Chunking Strategy
- **Size**: 200-500 tokens per chunk (too small = lost context, too large = noise)
- **Overlap**: 10-20% overlap between chunks to maintain context
- **Boundaries**: Break at natural boundaries (paragraphs, sentences)

### Indexing
- Use HNSW index for most use cases
- Set `ef_search` parameter for speed/accuracy tradeoff
- Monitor index size and memory usage

### Query Optimization
- Adjust top-k (typically 5-10 for RAG)
- Use metadata filtering when possible
- Cache frequent queries
- Monitor query latency

## Performance Considerations

### Why Use Vector Database Instead of Local Calculation?

**BM25 (Keyword Search)**
- Sparse search based on term frequency
- Fast to compute locally (just counting terms)
- No indexing needed for small datasets
- We do this locally in our RAG system

**Vector Search (Dense Search)**
- High-dimensional similarity (768+ dimensions)
- Brute force: O(n) similarity calculations
- With HNSW index: O(log n) approximate search
- Qdrant's native search uses HNSW index on server

**Current Implementation Issues**
- Uses scroll to fetch ALL vectors from Qdrant (network overhead)
- Computes similarity locally with ALL vectors (O(n))
- No index utilization (slow for large datasets)

**With Qdrant Native Search**
- Uses HNSW index on server side
- Only computes similarity with candidate vectors
- Reduces network traffic (only returns top-k)
- Implemented in Rust (much faster than Python)

**Performance Comparison (10,000 vectors)**
- Current (scroll + local): ~100-500ms per query
- Qdrant native search: ~5-20ms per query

## Summary

Vector databases are essential for RAG systems because they enable efficient semantic search. While BM25 works well for keyword matching, vector search captures semantic meaning that keywords miss. For production RAG systems, using a proper vector database with indexing (like Qdrant with HNSW) is much faster than brute force similarity calculation.
